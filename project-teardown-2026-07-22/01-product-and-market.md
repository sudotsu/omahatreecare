# Product and Market

## Product thesis

The site differentiates by providing useful homeowner guidance before asking for contact information. The strongest version of that thesis is: give a truthful planning or screening result, state what the tool cannot establish, and provide an on-site next step without turning uncertainty into fear. The rebuilt cost planner fits that direction better than the previous archetype funnel.

## Benchmark selection

This cleanup is benchmarked against four relevant standards rather than a broad competitor scrape:

- the site's own published Treehouse pricing and evidence controls;
- Next.js 16.2 installed documentation and current patched framework releases;
- WCAG AA behavior represented by automated contrast, structure, names, and keyboard checks, with real assistive technology still required;
- a trustworthy local-service conversion path: useful information, consistent promises, transparent failure, and verified delivery.

Competitor price opacity is not a useful quality target. The appropriate ceiling is a planner that helps budgeting without presenting an online range as a quote.

## Current landscape evidence

Current primary dependency research was dated 2026-07-22. Next.js 16.2.11 is the available patched release for two advisories affecting the audited 16.2.6 version. The latest verified patch still declares a sharp range below sharp's separate security fix; current source does not expose untrusted image processing, so that residual risk should be tracked rather than concealed with an unsupported override.

Within the product, the tools hub, homepage, dynamic metadata, and individual tool implementations duplicate summaries and counts. This has already produced a visible contradiction: the hub says the species guide covers 12 trees while the implementation and homepage correctly describe 10.

## Differentiation, contradictions, and trajectory

The defensible differentiator is honesty plus usefulness, not specificity for its own sake. Current strengths support that positioning, but contradictions weaken it: accurate cost content sits beside stale README language; tool counts differ; accessibility outcomes are asserted more strongly than they are tested; and the primary request-delivery path is not operationally verified.

If the implementation sequence is followed, the site can become a strong local-service reference with unusually transparent tools. If warning cleanup is treated as the entire job, the more consequential runtime and trust defects will remain under a green build.

## Consequences of changing or retaining major directions

- Retain the bounded tools and shared pricing source; replacing them with a quote gate would discard the product's clearest differentiation.
- Separate layout ownership by route family; retaining stacked shells creates recurring semantic and styling defects.
- Extend evidence-backed publishing to legacy content; retaining two standards for articles invites unsupported claims back into search-facing pages.
- Keep the no-unsafe-eval CSP and configure client validation to fit it; weakening CSP would trade a diagnostic issue for a real security regression.
- Prefer supported framework upgrades over transitive overrides; a temporary override would require explicit owner approval and its own compatibility tests.
